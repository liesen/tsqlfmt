= List layout cleanup plan

This note sketches a cleanup of `src/tsqlfmt/Lists.fs` after the recent `OVER (...)` work.

The main issue is that the current API mixes two different concerns:

- sequence layout
- comma decoration

That worked for the initial helpers, but `OVER (...)` needed the same comma decoration with a different layout.
That is a sign that the current split is at the wrong boundary.

== Core idea

Split list formatting into three layers:

- sequence layout
- item decoration
- anchored convenience wrappers

This keeps the structure cleaner:

- layout answers where items go
- decoration answers where commas go
- anchoring answers what the list attaches to, such as `ORDER BY`

== Terminology

Use words that match the style model more closely:

- `layout` instead of `policy`
- `anchored` instead of `headed`
- `decoration` for comma placement and comma spacing

`anchored` is a better name than `headed` because the extra document is not the head of the list.
It is the anchor that the list attaches to.

== Rename map

- `SequencePolicy` -> `SequenceLayout`
- `headedSequenceDoc` -> `anchoredSequenceDoc`
- `listSequencePolicy` -> `listSequenceLayout`
- `ddlSequencePolicy` -> `ddlSequenceLayout`
- `commaItems` -> `commaDecoratedItems`
- `ddlCommaItems` -> `ddlCommaDecoratedItems`
- `headedCommaListDoc` -> `anchoredCommaSeparatedListDoc`
- `headedCommaListWithPolicyDoc` -> `anchoredCommaSeparatedListWithLayoutDoc`

== Proposed structure

=== Sequence layout

[source,fsharp]
----
type SequenceLayout =
    { placeFirstItemOnNewLine: bool
      firstItemIndent: int option
      subsequentItemsIndent: int option }

val sequenceDoc : SequenceLayout -> Doc list -> Doc
val anchoredSequenceDoc : SequenceLayout -> Doc -> Doc list -> Doc
val withFirstItemIndent : int -> SequenceLayout -> SequenceLayout
----

This layer owns only line placement and indentation.

=== Decoration

The lightest version of the cleanup keeps decoration as helper functions:

[source,fsharp]
----
val commaDecoratedItems : Style -> Doc list -> Doc list
val ddlCommaDecoratedItems : Doc list -> Doc list
----

If we need more than one or two decoration families later, we can make the decoration rules explicit:

[source,fsharp]
----
type CommaPlacement =
    | BeforeItems
    | AfterItems

type SequenceDecoration =
    { commaPlacement: CommaPlacement
      addSpaceBeforeComma: bool }

val commaDecoratedItems : SequenceDecoration -> Doc list -> Doc list
----

That step is optional for now.

=== Style-driven layouts and decorations

[source,fsharp]
----
val listSequenceLayout : Style -> SequenceLayout
val ddlSequenceLayout : Style -> SequenceLayout

val listCommaSeparatedDecoration : Style -> SequenceDecoration
----

These builders should use style-model terms such as:

- `placeFirstItemOnNewLine`
- `placeSubsequentItemsOnNewLines`
- `indentListItems`
- `alignSubsequentItemsWithFirstItem`
- `placeCommasBeforeItems`
- `addSpaceBeforeComma`

=== Thin convenience wrappers

[source,fsharp]
----
val commaListDoc : Style -> Doc list -> Doc
val anchoredCommaSeparatedListDoc : Style -> Doc -> Doc list -> Doc
val anchoredCommaSeparatedListWithLayoutDoc : SequenceLayout -> Style -> Doc -> Doc list -> Doc
val ddlCommaListDoc : Style -> Doc list -> Doc
----

These wrappers should stay thin.
They should compose decoration and layout rather than hiding both in one combined helper.

== Why this is cleaner

Today, `OVER (...)` needs:

- the same comma decoration as generic lists
- a different layout than generic lists

That is exactly the kind of case that should be handled by composition.

The intended composition is:

[source,fsharp]
----
items
|> commaDecoratedItems cfg
|> anchoredSequenceDoc overClauseLayout anchorDoc
----

Instead of introducing more specialized helpers, the formatter should be able to reuse:

- an existing decoration step
- an existing layout step
- a custom layout value

== `OVER (...)` example

The `OVER (...)` clause should look like an anchored comma-separated sequence with a custom layout.

Conceptually:

[source,fsharp]
----
let overClauseLayout cfg =
    { placeFirstItemOnNewLine = false
      firstItemIndent = None
      subsequentItemsIndent = Some(indentWidth cfg) }
----

Then a clause like `PARTITION BY` can be expressed as:

[source,fsharp]
----
partDocs
|> commaDecoratedItems cfg
|> anchoredSequenceDoc (overClauseLayout cfg) (keyword cfg "PARTITION" <++> keyword cfg "BY")
----

This is cleaner than a special-purpose `overClauseListDoc` because it says directly:

- use comma-separated decoration
- use an OVER-specific layout
- anchor the sequence to `PARTITION BY`

== Recommended migration order

. Rename the sequence terms:
  - `SequencePolicy` -> `SequenceLayout`
  - `headedSequenceDoc` -> `anchoredSequenceDoc`
  - `listSequencePolicy` -> `listSequenceLayout`
  - `ddlSequencePolicy` -> `ddlSequenceLayout`
. Rename the decoration helpers:
  - `commaItems` -> `commaDecoratedItems`
  - `ddlCommaItems` -> `ddlCommaDecoratedItems`
. Rename the convenience wrappers:
  - `headedCommaListDoc` -> `anchoredCommaSeparatedListDoc`
  - `headedCommaListWithPolicyDoc` -> `anchoredCommaSeparatedListWithLayoutDoc`
. Update call sites in `Formatter.fs` mechanically.
. In a later pass, decide whether `SequenceDecoration` should become an explicit type.

== Recommendation

The best near-term cleanup is:

- rename `policy` to `layout`
- rename `headed` to `anchored`
- keep decoration as explicit helper functions
- prefer composition of layout and decoration over more `...WithPolicyDoc` helpers

The best longer-term cleanup is:

- make decoration an explicit concept if a second or third decoration family appears

That keeps `Lists.fs` structured around the real axes of variation instead of around ad hoc convenience functions.

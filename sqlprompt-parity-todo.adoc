= SQL Prompt Parity Gaps

These are five important SQL Prompt parity gaps from the current fixture-backed parity suite.
The focus here is not priority order. The focus is what `tsqlfmt` does not yet implement in SQL Prompt-compatible form, and which style settings appear to drive the difference.

== 1. Leave-as-is casing when `--applyCasing` is absent

Current gap:: `tsqlfmt` still recases many structurally formatted nodes even when SQL Prompt compatibility expects original casing to be preserved.

Observed in:: `tests/tsqlfmt.Tests/TestData/test1.actual.sql`, `tests/tsqlfmt.Tests/TestData/test13.actual.sql`, `tests/tsqlfmt.Tests/TestData/test21.actual.sql`, `tests/tsqlfmt.Tests/TestData/test28.actual.sql`, `tests/tsqlfmt.Tests/TestData/test30.actual.sql`

SQL Prompt does:: Preserve most original keyword, function, and type casing when `--applyCasing` is not passed.

tsqlfmt does:: Recase many structurally formatted nodes to uppercase anyway.

Related settings:: `casing.reservedKeywords`, `casing.builtInFunctions`, `casing.builtInDataTypes`, and the SQL Prompt-compatible `--applyCasing` CLI switch.

Example::
+
[cols="a,a",options="header"]
|===
|SQL Prompt |tsqlfmt

a|
[source,sql]
----
SELECT Cast(pos.upload_date as date) as 'Date'
----

a|
[source,sql]
----
SELECT CAST(pos.upload_date AS DATE) AS 'Date'
----
|===

== 2. Multiline select-list indentation and hanging continuation policy

Current gap:: `tsqlfmt` does not currently implement SQL Prompt-style first-item alignment for multiline select lists, and more generally keeps using its own simpler continuation layout.

Observed in:: `tests/tsqlfmt.Tests/TestData/test5.actual.sql`, `tests/tsqlfmt.Tests/TestData/test6.actual.sql`, `tests/tsqlfmt.Tests/TestData/test8.actual.sql`, `tests/tsqlfmt.Tests/TestData/test16.actual.sql`, `tests/tsqlfmt.Tests/TestData/test19.actual.sql`

SQL Prompt does:: Align continuation items deeper, typically under the first item rather than at one fixed indent level.

tsqlfmt does:: Use a simpler fixed-indent continuation layout.

Related settings:: `lists.alignSubsequentItemsWithFirstItem`, `lists.alignItemsAcrossClauses`, and, for related continuation behavior, `operators.andOr.alignment`.

Example::
+
[cols="a,a",options="header"]
|===
|SQL Prompt |tsqlfmt

a|
[source,sql]
----
SELECT id,
       name,
       a,
       b
----

a|
[source,sql]
----
SELECT id,
    name,
    a,
    b
----
|===

== 3. Routine header and object-body layout for procedures, functions, views, and tables

Current gap:: `tsqlfmt` does not yet format several routine and object headers in the same shape as SQL Prompt, especially around opening parentheses, parameter lists, `RETURNS`, and `AS`.

Observed in:: `tests/tsqlfmt.Tests/TestData/test2.actual.sql`, `tests/tsqlfmt.Tests/TestData/test22.actual.sql`, `tests/tsqlfmt.Tests/TestData/test25.actual.sql`, `tests/tsqlfmt.Tests/TestData/test26.actual.sql`, `tests/tsqlfmt.Tests/TestData/test27.actual.sql`

SQL Prompt does:: Prefer line breaks before parameter lists, `RETURNS`, `AS`, and opening parentheses in several routine/object forms.

tsqlfmt does:: Keep more of the header on one line.

Related settings:: Mostly `ddl.parenthesisStyle`, `ddl.indentParenthesesContents`, `ddl.collapseShortStatements`, `ddl.collapseStatementsShorterThan`, plus construct-specific formatter logic. There is no single dedicated routine-header setting today.

Example::
+
[cols="a,a",options="header"]
|===
|SQL Prompt |tsqlfmt

a|
[source,sql]
----
create function dbo.fn_test26
()
returns table
as
----

a|
[source,sql]
----
CREATE FUNCTION dbo.fn_test26()
RETURNS TABLE
AS
----
|===

== 4. Insert column and values layout, including parenthesized clause placement

Current gap:: `tsqlfmt` does not yet follow SQL Prompt's more expanded insert-column and values layout in several common cases.

Observed in:: `tests/tsqlfmt.Tests/TestData/test9.actual.sql`, `tests/tsqlfmt.Tests/TestData/test10.actual.sql`, `tests/tsqlfmt.Tests/TestData/test20.actual.sql`

SQL Prompt does:: Break column lists and value tuples onto dedicated lines more aggressively.

tsqlfmt does:: Keep `INSERT INTO ... (...) VALUES ...` flatter.

Related settings:: `insertStatements.columns.parenthesisStyle`, `insertStatements.columns.placeSubsequentColumnsOnNewLines`, `insertStatements.values.parenthesisStyle`, `insertStatements.values.indentContents`, `insertStatements.values.placeSubsequentValuesOnNewLines`.

Example::
+
[cols="a,a",options="header"]
|===
|SQL Prompt |tsqlfmt

a|
[source,sql]
----
INSERT INTO users
(
    id,
    name,
    email
)
VALUES (1, 'Alice', 'alice@example.com');
----

a|
[source,sql]
----
INSERT INTO users (id, name, email)
VALUES (1, 'Alice', 'alice@example.com');
----
|===

== 5. Parenthesized subquery and boolean-condition collapse choices

Current gap:: `tsqlfmt` does not yet match SQL Prompt's collapse choices for some short parenthesized subqueries and some boolean-condition continuation shapes.

Observed in:: `tests/tsqlfmt.Tests/TestData/test24.actual.sql`, `tests/tsqlfmt.Tests/TestData/test4.actual.sql`, `tests/tsqlfmt.Tests/TestData/test11.actual.sql`, `tests/tsqlfmt.Tests/TestData/test29.actual.sql`

SQL Prompt does:: Keep some `EXISTS (SELECT ...)` and similar condition forms collapsed, and uses different boolean continuation alignment.

tsqlfmt does:: Expand some parenthesized subqueries and use our current boolean continuation rules.

Related settings:: `dml.collapseShortSubqueries`, `dml.collapseSubqueriesShorterThan`, `parentheses.collapseShortParenthesisContents`, `parentheses.collapseParenthesesShorterThan`, and `operators.andOr.alignment`.

Example::
+
[cols="a,a",options="header"]
|===
|SQL Prompt |tsqlfmt

a|
[source,sql]
----
IF EXISTS (SELECT 1 FROM sys.tables WHERE name = 'test24')
    SET @sql = 'DROP TABLE test24'
----

a|
[source,sql]
----
IF EXISTS (
    SELECT 1
    FROM sys.tables
    WHERE name = 'test24'
)
    SET @sql = 'DROP TABLE test24'
----
|===

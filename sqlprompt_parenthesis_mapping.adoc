= SQL Prompt parenthesis mapping

This note maps SQL Prompt style concepts for parentheses to the matching `tsqlfmt` style concepts and public combinators in `src/tsqlfmt/Parenthesis.fs`.

The guiding rule is simple: SQL Prompt concept and `tsqlfmt` concept should match when the concept is user-facing in style JSON.
We do not need to copy every lower-level formatting distinction from SQL Prompt.

== Scope

This document focuses on parenthesis placement and indentation concepts.
It does not try to catalog every place where literal `(` and `)` appear in SQL syntax.

== Current `tsqlfmt` combinators

`tsqlfmt` currently exposes these public parenthesis combinators:

- `expressionParensDoc`
- `ddlParensDoc`
- `cteBodyParensDoc`
- `cteColumnListParensDoc`
- `insertColumnListParensDoc`
- `insertValuesListParensDoc`
- `functionCallParensDoc`

These are style-bucket combinators.
They are not meant to mirror every AST node that happens to contain parentheses.

== Core principle

When SQL Prompt exposes a distinct style concept in style JSON, `tsqlfmt` should usually expose the same concept.

When SQL Prompt does not expose a distinct style concept, `tsqlfmt` should usually reuse an existing concept instead of inventing a new public bucket.

This gives two practical rules:

- match SQL Prompt at the style-concept level
- do not copy SQL Prompt distinctions that are not visible in style JSON

== Design rule

Parenthesis handling follows style concepts, not AST nodes.

A public parenthesis combinator should exist only when there is a real
user-facing style concept behind it. If SQL Prompt and `tsqlfmt` expose the
same concept in style JSON, `tsqlfmt` should usually expose the same public
bucket.

This leads to four rules:

- Public combinators represent style buckets.
- A small private helper implements shared parenthesis mechanics.
- If no separate style concept exists, reuse an existing bucket instead of
  inventing a new one.
- `group` belongs around the semantic unit, not just around the parentheses.

Examples:

- generic expressions -> `expressionParensDoc`
- DDL definition lists -> `ddlParensDoc`
- CTE body -> `cteBodyParensDoc`
- CTE column list -> `cteColumnListParensDoc`
- insert values -> `insertValuesListParensDoc`
- function calls -> `functionCallParensDoc`
- `IN (value, ...)` -> `inValuesParensDoc`
- `OVER (...)` -> `expressionParensDoc`

`OVER (...)` is a good example of the last rule. The parentheses use the
generic bucket, but the important grouping boundary is the full windowed
expression, such as `SUM(x) OVER (...)`, not the parenthesized clause alone.

== What `tsqlfmt` already matches well

- generic expressions and subqueries -> `expressionParensDoc`
- DDL definition-style lists and view column lists -> `ddlParensDoc`
- CTE body -> `cteBodyParensDoc`
- CTE column list -> `cteColumnListParensDoc`
- insert columns -> `insertColumnListParensDoc`
- insert row values -> `insertValuesListParensDoc`
- function call arguments -> `functionCallParensDoc`

== Where `tsqlfmt` still differs or is incomplete

- `IN (value, ...)`
  - SQL Prompt and `tsqlfmt` already share the same style concept in JSON under `operators.in.*`
  - `tsqlfmt` still formats this manually in `src/tsqlfmt/Formatter.fs`
  - the natural next step is a dedicated `inValuesParensDoc`

- `OVER (...)`
  - SQL Prompt treats this separately
  - but there is no separate style-json concept for over-clause parentheses
  - `tsqlfmt` should prefer the symmetric public concept: generic expression parentheses

- routine parameter lists
  - we have not yet traced the exact SQL Prompt style concept cleanly enough
  - until that is clear, they should stay manual rather than being forced into the wrong bucket

== Recommended taxonomy

The current public taxonomy is coherent:

- `expressionParensDoc`
- `ddlParensDoc`
- `cteBodyParensDoc`
- `cteColumnListParensDoc`
- `insertColumnListParensDoc`
- `insertValuesListParensDoc`
- `functionCallParensDoc`

The next likely addition is:

- `inValuesParensDoc`

The important negative rule is:

- do not add a new public combinator unless it corresponds to a real style concept in JSON

That is why `OVER (...)` should probably reuse `expressionParensDoc` instead of getting its own public combinator.

== Fundamental combinator

Below the public style-bucket combinators, `Parenthesis.fs` still wants one shared structural primitive.

The shape is:

- opening `(`
- contents with optional indentation
- closing `)`
- a style choice that controls line breaking

A reasonable lower-level shape is:

[source,fsharp]
----
type ParenthesisLayoutSpec =
    {
        contentsIndent: int option
        style: ParenthesisStyle
    }

val parenthesize : ParenthesisLayoutSpec -> Doc -> Doc
----

The public combinators should remain the semantic entry points.
The low-level primitive should stay style-parameterized, not AST-parameterized.

== Next step

The clearest remaining parenthesis concept that is present in style JSON but not yet mapped to a public `Parenthesis` combinator is:

- `operators.in.*` for `IN (value, ...)`
